**Risk:** SQL injection via `req.query.sort`. Attacker can append arbitrary SQL (e.g., `id; DROP TABLE invoices--` or subqueries via UNION). `ORDER BY` cannot be parameterized in most drivers, so prepared statements alone don't fix it.

Secondary: information disclosure via error-based injection, DoS via expensive sorts, and broken pagination/logic.

**Mitigation:** Whitelist allowed columns and directions.

```js
const allowedCols = ['id', 'date', 'amount', 'customer_id'];
const allowedDirs = ['ASC', 'DESC'];

const { sort = 'id', dir = 'ASC' } = req.query;
if (!allowedCols.includes(sort) || !allowedDirs.includes(String(dir).toUpperCase())) {
  return res.status(400).send('Invalid sort');
}
const sql = `SELECT * FROM invoices ORDER BY ${sort} ${dir.toUpperCase()}`;
```

Also: replace `SELECT *` with explicit columns, add authz checks, index sort columns, enforce pagination (`LIMIT`/`OFFSET` with bounds).

2. Static analysis: semgrep/CodeQL rule for string-concatenated SQL (`js/sql-injection`).
3. Dynamic: sqlmap against the endpoint — expect no injection reported.
4. DB logs: confirm only queries with whitelisted identifiers reach the DB.
